**A) Create `src/__tests__/security/jwt.test.ts` — JWT edge case tests per RESEARCH.md lines 506-577 and D-53**

Import: `describe, it, expect, beforeAll, afterAll` from vitest; `request` from supertest; `jwt` from jsonwebtoken; `app` from `../../app`; `createTestDb, seedTestData` from `../setup`; `signToken` from `../../lib/jwt`

Test cases (8 scenarios from RESEARCH.md):

1. **rejects request with no Authorization header:**
   - `GET /api/dashboard?start=2026-06-01&end=2026-06-26` WITHOUT Authorization header
   - Assert: 401, `res.body.error.code === 'UNAUTHORIZED'`

2. **rejects malformed Bearer token (not 3-part JWT):**
   - Set `Authorization: Bearer invalid.jwt.token` (2 dots but invalid base64)
   - Assert: 401, `res.body.error.code === 'UNAUTHORIZED'`

3. **rejects expired JWT token:**
   - `jwt.sign({ userId: 'test', outletId: 'test-outlet' }, TEST_SECRET, { expiresIn: '0s' })`
   - Wait 1100ms (ensure expiry passes)
   - Assert: 401

4. **rejects JWT signed with different secret:**
   - `jwt.sign({ userId: 'test', outletId: 'test-outlet' }, 'wrong-secret-key', { expiresIn: '1h' })`
   - Assert: 401

5. **rejects tampered JWT payload (signature mismatch):**
   - Generate valid token, then modify the payload portion:
     - `const parts = token.split('.')`
     - `const tamperedPayload = Buffer.from(JSON.stringify({ userId: 'hacker', outletId: 'other-outlet' })).toString('base64url')`
     - `const tamperedToken = parts[0] + '.' + tamperedPayload + '.' + parts[2]`
   - Assert: 401

6. **rejects JWT with missing required claims:**
   - `jwt.sign({ userId: 'test' }, TEST_SECRET, { expiresIn: '1h' })` — missing outletId
   - Note: verifyToken returns decoded payload; authMiddleware sets req.user. If the downstream code destructures `outletId`, this may cause a runtime error rather than 401. Test to see current behavior and document.
   - If it passes through authMiddleware, flag as a finding (Major — should validate payload shape in authMiddleware or signToken)

7. **rejects JWT with algorithm 'none':**
   - `jwt.sign({ userId: 'test', outletId: 'test-outlet' }, TEST_SECRET, { algorithm: 'none', expiresIn: '1h' })`
   - Note: jsonwebtoken v9+ should reject this. Document behavior.
   - Assert: the token should be rejected (either sign fails, or verify fails)

8. **valid JWT passes authentication and returns 200:**
   - Use `signToken({ userId: 'test-owner-1', outletId: 'test-outlet-1' })` with correct secret
   - Assert: 200, `res.body.success === true` for GET /api/dashboard (positive control — proves infrastructure works)

**Use `TEST_SECRET` for all test JWTs:**
- Read from `process.env.JWT_SECRET` (set in vitest.config.ts env from Plan 04-01: `'test-secret-key-for-phase04'`)
- Fallback: use the same fallback as production: `'fallback-secret-not-for-production'`

Test that SQL injection strings in query params do NOT affect query behavior:

   - Assert: 400 (Zod date format validation rejects non-YYYY-MM-DD)
   - If 200 (bypass), flag as Critical

2. **SQL injection in sales body:**
   - Send `POST /api/sales` with `{ date: '2026-06-26', revenue: 500000, top_menu_items: ["Nasi Goreng' OR '1'='1"] }`
   - Assert: 201 (menu items are stored as JSON string, SQL injection in field value is harmless with Prisma parameterized queries)
   - Verify: the menu item is stored exactly as the injected string (no SQL executed)

3. **SQL injection in report date params:**
   - Send `GET /api/report?start=2026-06-01'; DROP TABLE DailySales;--&end=2026-06-26`
   - Assert: 400 (Zod validation rejects format)

**Part 2: XSS injection via sales body (D-53)**

4. **XSS payload in menu items:**
   - Send `POST /api/sales` with `{ date: '2026-06-27', revenue: 500000, top_menu_items: ["<script>alert('xss')</script>"] }`
   - Assert: 201 (stored as string)
   - Verify: the value is stored as-is; React's JSX auto-escaping handles rendering. Check: grep frontend for `dangerouslySetInnerHTML` — should be zero.

**Part 3: CSV injection verification (D-53)**

Per RESEARCH.md lines 582-631: unit-test the escapeCell logic. Since escapeCell is in `frontend/src/lib/csvGenerator.ts` (client-side), create the test in the backend test file but test a replicated/imported version of the logic. Better: create a standalone unit test.

5. **CSV formula injection prefixes neutralized:**
   - Replicate the `escapeCell` logic in the test (or import the logic pattern for verification)
   - Test inputs: `'=1+2'` → output starts with `\t`
   - Test: `'+SUM(A1:A10)'` → starts with `\t`
   - Test: `'-A1+A2'` → starts with `\t`
   - Test: `'@SUM(A1:A10)'` → starts with `\t`
   - Test: `'Normal Text'` → does NOT start with `\t`
   - Test: `'12345'` → does NOT start with `\t`
   - Test: `'=1+2";DROP TABLE'` → starts with `\t` AND contains `"` (RFC 4180 quoting still applied after tab prefix)
