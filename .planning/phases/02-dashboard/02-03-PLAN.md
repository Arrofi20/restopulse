  <action>
Create `frontend/src/contexts/AuthContext.tsx` — React Context for JWT token management and auth state (per D-17, D-18 user identity display).

**Interface:**
```typescript
interface AuthState {
  token: string | null;
  user: { id: string; username: string } | null;
  isAuthenticated: boolean;
  isLoading: boolean;
}

interface AuthContextType extends AuthState {
  login: (username: string, password: string) => Promise<void>;
  logout: () => void;
}
```

**AuthProvider component:**
- Initialize state: `token` from `localStorage.getItem('restopulse_token')`, `isLoading: false`, `isAuthenticated` based on token presence.
- On mount: if token exists, call `GET /api/auth/logout` is NOT needed for validation. Just trust the stored token (it will be validated by the API on next request; 401 redirects to login via apiClient).
- `login(username, password)`:
  - Call `post<{ success: boolean; data: { token: string; owner: { id: string; username: string } } }>('/auth/login', { username, password })`.
  - On success: call `setToken(result.data.token)`, set state `{ token: result.data.token, user: result.data.owner, isAuthenticated: true }`.
  - On error: re-throw so LoginPage can display the error message.
- `logout()`:
  - Call `clearToken()`, reset state to `{ token: null, user: null, isAuthenticated: false }`.
  - Navigate to `/login` via `window.location.href = '/login'` (simple for v1).
- Context value: `{ token, user, isAuthenticated, isLoading, login, logout }`.

**Modify `frontend/src/main.tsx`:**
- Wrap `<App />` in `<AuthProvider>`:
```tsx
import { AuthProvider } from './contexts/AuthContext';
// ...
createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <AuthProvider>
      <App />
    </AuthProvider>
  </StrictMode>
);
```

DO NOT use cookies or httpOnly — the backend uses Bearer tokens (per RESEARCH.md Pitfall 2). DO NOT decode JWT client-side (unnecessary for v1). DO implement the 401 auto-logout behavior; apiClient already calls `clearToken()` and redirects on 401 — the AuthContext should listen for token removal and update state accordingly (use `storage` event listener for cross-tab sync).
</action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Login form → API | User credentials submitted via POST /api/auth/login |
| localStorage | Bearer token persisted client-side |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-10 | Spoofing | LoginPage form | medium | mitigate | Backend rate-limits auth routes (5 req/15min from Phase 1); credentials validated with bcrypt |
| T-02-11 | Information Disclosure | localStorage token | medium | accept | Token is XSS-readable (v1 MVP risk); 24h expiry limits window; React's default escaping prevents stored XSS |
| T-02-12 | Elevation of Privilege | ProtectedRoute | medium | mitigate | Unauthenticated users redirected to /login; token validated by backend on every API call |
| T-02-13 | Tampering | Client-side routing | low | accept | react-router-dom handles routing; no server-side rendering — no SSR injection vector |
</threat_model>
