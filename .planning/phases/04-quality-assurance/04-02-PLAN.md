**Create `k6/` directory and `k6/load-test.js`** — JavaScript file for the Grafana k6 load testing tool (NOT Node.js, NOT the npm k6 dummy package).

**Script structure (per RESEARCH.md Pattern 4):**

1. **Options block:**
   - `vus: 50` (50 concurrent virtual users per D-43)
   - `duration: '30s'` (30-second test run)
   - `thresholds`:
     - `http_req_duration: ['p(95)<500']` — 95th percentile of ALL requests must be under 500ms (NFR §9.4)
     - `http_req_failed: ['rate<0.01']` — error rate must be under 1%
     - `'http_req_duration{endpoint:sales}': ['p(95)<500']` — sales endpoint per-tag threshold
     - `'http_req_duration{endpoint:dashboard}': ['p(95)<400']` — dashboard reads should be faster (CQRS-lite pre-computed SalesTrend)
     - `'http_req_duration{endpoint:report}': ['p(95)<500']` — report endpoint threshold

2. **Setup function (`setup()`):**
   - The `BASE_URL` constant: `'http://localhost:3000'` (Express backend on default port)
   - Use `http.post()` to call `POST /api/auth/login` with credentials `{ username: 'testuser', password: 'testpass123' }` to obtain a valid JWT
   - Return `{ token: res.json('data.token') }` so the default function receives a fresh token
   - This makes the test self-contained — no hardcoded tokens (RESEARCH Open Question #2)

3. **Default function:**
   - Accept `data` parameter (the token from setup)
   - `const rand = Math.random()`
   - **40% probability — POST /api/sales:**
     - Random revenue: `Math.floor(Math.random() * 1000000) + 100000`
     - Random menu items: pick 1-3 from a small array `['Nasi Goreng', 'Mie Ayam', 'Es Teh', 'Ayam Bakar', 'Soto']`
     - Today's date: generate dynamically using JS Date
     - Send with `Content-Type: application/json` and `Authorization: Bearer ${data.token}`
     - Tag: `{ endpoint: 'sales' }`
     - Check: status is 201 (first sale for today) or 409 (duplicate — acceptable under load)
       - Use `check(res, { 'sales_ok': (r) => r.status === 201 || r.status === 409 })`
   - **30% probability — GET /api/dashboard:**
     - Query: `?start=2026-06-01&end=2026-06-26`
     - Tag: `{ endpoint: 'dashboard' }`
     - Check: status is 200
   - **30% probability — GET /api/report:**
     - Query: `?start=2026-06-01&end=2026-06-26`
     - Tag: `{ endpoint: 'report' }`
     - Check: status is 200
   - `sleep(1)` between iterations (1 second think time per VU)

4. **Database index verification (D-45):**
   - The k6 script itself does NOT modify indexes. After Task 1 runs the load test:
     - If p(95) is ≤500ms: NO index changes needed — D-45 satisfied
     - If p(95) > 500ms: document the failing benchmark in SUMMARY.md and add a Prisma migration with additional indexes (e.g., on `DailySales(date)`, `SalesTrend(date)`) — but this is done as a follow-up, not in this task
   - Add a comment at the top of `k6/load-test.js` documenting the current state: `// D-45: Existing indexes [outlet_id, date] on DailySales + SalesTrend. Only add indexes if p(95) > 500ms.`

**Running the script (record results in SUMMARY.md):**
- The agent runs: `k6 run k6/load-test.js` (NOT `node k6/load-test.js`)
- Record: p(95) values for each endpoint, total requests, error rate, whether thresholds passed
- If k6 is not installed, the agent creates a checkpoint:human-verify asking the user to install k6 via `winget install k6 --source winget`

**Important: Do NOT install the npm `k6` dummy package.**

       --output-path=lighthouse/reports/{name} \
       --chrome-flags="--headless" \
       --only-categories=performance,accessibility \
       --throttling-method=devtools \
       --throttling.requestLatencyMs=150 \
       --throttling.downloadThroughputKbps=1600 \
       --throttling.uploadThroughputKbps=750 \
       --throttling.cpuSlowdownMultiplier=4
