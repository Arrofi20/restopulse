<objective>
Make the codebase environment-aware so it can run against SQLite in local development and PostgreSQL in production on Render. Update Prisma schema, Prisma config, server startup validation, and package scripts, then sync the production schema.

Purpose: Without env-based datasource configuration, the production Render PostgreSQL database cannot be targeted, blocking all deployment. This plan implements D-60.
Output: Env-aware Prisma setup, documented environment variables, and a production-ready migration command.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| env file -> runtime | Untrusted environment variables cross into application config |
| local dev -> production | Same codebase targets two database providers; misconfiguration risks data loss |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-01 | Information Disclosure | .env.example | low | accept | .env.example contains only placeholder secrets; real secrets live in Render dashboard |
| T-05-02 | Tampering | prisma/schema.prisma provider | high | mitigate | Validate DB_PROVIDER is exactly "sqlite" or "postgresql" at server startup; reject unknown providers |
| T-05-03 | Denial of Service | npx prisma migrate deploy | medium | mitigate | Run against production DATABASE_URL only; never default to local dev DB for deploy |
</threat_model>
