  <action>
    Create ReportRepository.ts that queries both SalesTrend (for daily revenue and menu popularity) and DailySales (for transaction count per day) within a date range scoped to outlet_id.

    - Constructor accepts no args and uses the existing `prisma` singleton import pattern (mirrors SalesTrendRepository).
    - Method `getReportData(outlet_id: string, start: Date, end: Date)`:
      1. Query SalesTrend rows via `prisma.salesTrend.findMany` where outlet_id and date gte/lte, orderBy date asc.
      2. Query DailySales rows via `prisma.dailySales.findMany` for the same outlet_id and date range, orderBy date asc.
      3. Map SalesTrend rows to objects with `date` (YYYY-MM-DD string), `revenue` (number), `topMenu` (string — the first item's name from parsed menu_popularity JSON, or "-" if empty), `transactionCount` (number from matching DailySales row by date, or 0 if no match).
      4. For the top menu items across the entire period: aggregate all parsed menu_popularity.items by name summing count, sort desc, take top 3 names.
      5. Return `{ rows: ReportRow[], topItems: string[] }`.
    - Export the class. Add to `src/repositories/index.ts`.
    - Do NOT create a separate DailySalesRepository instance — use prisma directly in this repository (per D-33b agent discretion: live query preferred).
  </action>

<task type="auto">
  <name>Task 2: Create ReportService with date validation and aggregation</name>
  <files>src/services/ReportService.ts</files>
  <read_first>src/services/DashboardService.ts, src/validation/sales.schema.ts</read_first>
  <action>
    Create ReportService.ts that validates inputs, orchestrates repository queries, and formats the report response.

    - Constructor accepts `reportRepo: ReportRepository`.
    - Method `getReport(outlet_id: string, start: string, end: string)`:
      1. Validate `start` and `end` with the existing `dateRangeSchema` from `../validation/sales.schema` (DRY reuse per Phase 2 pattern).
      2. Parse UTC day boundaries: `startDate = new Date(start + 'T00:00:00Z')`, `endDate = new Date(end + 'T23:59:59.999Z')`.
      3. Call `reportRepo.getReportData(outlet_id, startDate, endDate)` to get rows and topItems.
      4. Compute summary: `totalRevenue = rows.reduce((sum, r) => sum + r.revenue, 0)`, `transactionCount = rows.reduce((sum, r) => sum + r.transactionCount, 0)`.
      5. Resolve outlet name via `prisma.outlet.findUnique({ where: { id: outlet_id }, select: { name: true } })` (same pattern as DashboardService).
      6. Return `{ outlet: { name }, period: { start, end }, summary: { totalRevenue, transactionCount, topItems }, rows }`.
    - Per D-31, read from SalesTrend (summary) + DailySales (detailed breakdown). Per D-33b, use live queries — no caching layer.
  </action>

    <automated>npx tsc --noEmit --skipLibCheck</automated>
  </verify>
  <done>ReportService.ts compiles. getReport validates date range via dateRangeSchema, queries ReportRepository, computes summary aggregates, resolves outlet name, and returns structured report data.</done>
